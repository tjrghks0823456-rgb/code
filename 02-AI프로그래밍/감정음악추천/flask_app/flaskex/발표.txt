★ 교수님 질문 대비 예상 Q&A (스마트 창고 물류 시스템)
1. 시스템 구조 관련 질문
Q1. 왜 ‘웹–HMI–PLC’ 구조로 만들었나요? 웹에서 PLC를 직접 제어하지 않은 이유가 있나요?

A1. PLC는 실시간성이 매우 중요한 장비라,
웹과 직접 연결하면 네트워크 지연·보안·안정성 문제를 해결하기 어렵습니다.
HMI(C# WinForms)는 Windows 환경에서 ADS 통신으로 PLC 표시 메모리를 직접 접근하므로
지연이 짧고 안정성이 높아 중간 제어기로 사용했습니다.

또한 HMI가 중간에서

데이터 보정

장비 이벤트 처리

불량값 필터링
등을 수행하여 서버로 전달하기 때문에
장비 보호와 데이터 신뢰성 확보에 적합합니다.

Q2. Flask 서버를 선택한 이유는 무엇인가요?

A2. Flask는

REST API 구축 용이

라우팅 구조 단순

센서 기반 JSON 교환에 최적화
되어 있습니다.

특히 프로젝트가 실시간 제어 중심이라
가벼운 서버 프레임워크가 효율적이었습니다.

Q3. polling 구조인데 WebSocket을 쓰지 않은 이유는 무엇인가요?

A3. PCST 장비의 센서 업데이트 주기가 1초 이상이고
실시간 동기화의 필요성이 크지 않기 때문에
3초 polling 기반 구조로도 성능적으로 충분했습니다.
또한 Flask 기본 구조가 WebSocket보다 REST에 적합했기 때문에
프로젝트 규모에 맞춰 polling 방식을 선택했습니다.

향후에는 WebSocket + PLC Event Task 구조로 확장할 계획입니다.

2. PLC·ADS 통신 관련 질문
Q4. TwinCAT ADS 통신은 정확히 어떤 역할을 하나요?

A4. ADS는 Beckhoff PLC가 사용하는 통신 프로토콜로,
PLC 변수(GVL, VAR_GLOBAL)를 메모리 주소 기반으로 직접 Read/Write할 수 있습니다.

이를 통해 C# HMI는

온습도 값 읽기

이벤트 플래그 읽기

램프/모터 상태 쓰기
같은 장비 제어를 안정적으로 수행할 수 있습니다.

Q5. PLC에서 센서값을 가져올 때 어떤 보정이 필요한가요?

A5. PCST Trainer의 센서 출력은 Raw 값이기 때문에
그대로 UI에 표시하면 실제 물리량과 차이가 납니다.

그래서 다음과 같은 선형 보정식을 사용했습니다.

온도 = a * Raw + b
습도 = c * Raw + d


값이 튀는 문제는 C#에서 1차 필터링(±15% 이상 변화 제한)을 적용해 해결했습니다.

Q6. PLC Task 주기 설정은 어떻게 했나요?

A6. PLC에서는 100ms Task 주기를 사용하여
센서 데이터를 백그라운드에서 지속적으로 Sampling하도록 구성했습니다.
이후 C# HMI는 1초 단위로 Polling하여
UI에 과도한 업데이트가 발생하지 않도록 조율했습니다.

3. 이벤트 처리 관련 질문
Q7. 지진/정전 이벤트 발생 조건은 어떻게 판단하나요?

A7.

지진 이벤트: 진동 센서 Raw 값이 특정 임계치(Threshold) 초과 시 이벤트 발생

정전 이벤트: PCST Trainer의 전원 플래그 OFF 감지

온도 이상: ΔT(변화량) 기반 → 1초 동안 급격한 상승/하강이면 이벤트 발생

이렇게 3가지 이벤트를 PLC 단계에서 감지하고
HMI → Flask → 관리자 웹으로 전달됩니다.

Q8. 정상화(Reset) 버튼은 어떤 과정을 거치나요?

A8. 정상화 버튼을 누르면 다음 흐름이 자동 실행됩니다.

PLC 이벤트 플래그 Reset

HMI 이벤트 상태 Update

서버 이벤트 로그 저장

관리자 Dashboard UI 갱신

즉 PLC → HMI → 서버 → 웹 전체가 동시에 동기화됩니다.

4. UI/UX 및 사용자 기능 관련 질문
Q9. 사용자와 관리자를 왜 분리된 UI로 구성했나요?

A9. 사용자와 관리자의 목적이 완전히 다르기 때문입니다.

사용자는: 보관 예약 / 물건 꺼내기 같은 단일 목적 중심

관리자는: 모니터링 / 승인·거절 / 이벤트 대응

따라서 UX 혼선을 줄이고 정확도를 확보하기 위해
UI 전체를 두 레이어로 완전히 분리했습니다.

Q10. 사용자 물건 예약 기능의 날짜 검증 방식은 어떤가요?

A10.

오늘 이전 날짜는 비활성

3일 이상 예약 시 관리자 승인 필수

냉동/냉장 선택 시 저장 가능한 물품인지 필터링

서버에서 추가 검증(Validation) 진행

즉, UI에서도 검증하고 서버에서도 이중으로 검증해
오입력 방지를 설계했습니다.

5. 로그·데이터 분석 관련 질문
Q11. 이벤트 로그와 사용자 로그를 별도로 관리한 이유는?

A11. 목적이 다르기 때문입니다.

이벤트 로그: 센서·장비 이상 상태를 분석

사용자 로그: 행동 이력(입고/출고 요청)을 분석

관리자 로그: 승인/거절 근거 추적

이렇게 분리하면
문제 발생 시 정확히 어디에서 문제가 발생했는지
Root Cause 분석이 훨씬 쉬워집니다.

Q12. 로그가 UI에 자동으로 갱신되는 방식은?

A12.

서버에서 로그를 JSON 배열로 제공

웹에서는 3초 polling으로 갱신

관리자 화면은 CSS Highlight로 신규 로그 강조

향후 WebSocket 적용 시 즉시 반영도 가능해집니다.

★ 1. 교수님 질문 대비 초고난도 심화 Q&A 세트
■ 통신/네트워킹 분야 심화 질문
Q1. ADS 통신은 메모리 직접 접근인데, Race Condition 문제는 어떻게 방지했나요?

A. TwinCAT PLC는 Task 주기가 고정되어 있어
메모리 갱신이 atomic하게 실행되며
ADS Read/Write는 단일 요청 단위로 처리되어
경쟁 상태가 발생할 가능성이 낮습니다.
추가적으로 C# 측에서는 1초 polling 주기로 처리하여
동일 변수에 대해 연속적인 Write가 겹치지 않도록 했습니다.

Q2. PLC 내부에서 Task 우선순위(priority)를 어떻게 설정했나?

A. PCST 센서 수집 Task는 100ms 주기 Standard Task로 설정했고,
모드 제어 Task는 이보다 낮은 우선순위를 주어
센서 안정성이 더 중요하도록 구성했습니다.

Q3. PLC 내부에서 온도 PID를 구현하지 않고 시뮬레이션 방식으로 구현한 이유는?

A. PCST Trainer는 실제 냉각장치가 없기 때문에
PID 제어 효과가 물리적으로 반영되지 않습니다.
따라서 저희는 PID를 모사하는 1차 지연(Lag) 모델 기반 시뮬레이션을 구현하여
온도 변화가 선형적으로 조절되도록 했습니다.

Q4. Flask가 동시 요청을 처리하기 어려운데, 병렬성 문제는 어떻게 해결했나요?

A. Flask는 기본적으로 단일 스레드 기반이지만
WSGI 서버인 Gunicorn을 사용하면 멀티워커로 확장할 수 있습니다.
저희 프로젝트는 실시간성이 낮은 polling 방식이기 때문에
병목이 발생하지 않는 구조입니다.

Q5. 왜 FastAPI가 아닌 Flask를 선택했나?

A. FastAPI가 비동기 성능 측면에서 뛰어나지만
본 프로젝트의 데이터량/요청량은 낮기 때문에
경량 + 구현 속도를 고려해 Flask를 선택했습니다.

■ 데이터 처리/보정 알고리즘 관련 질문
Q6. 센서 값이 순간적으로 튀는 경우 어떻게 처리했나요?

A. C# HMI에서
IQR 기반 이상치 제거 + 이동평균(3-point) 필터를 적용했습니다.

Q7. 보정식에서 선형 변환 대신 비선형 변환을 고려하지 않은 이유?

A. PCST Trainer의 온습도 센서는 실제 물리 센서가 아니라
가상 센서 기반 ADC 값이기 때문에
출력이 선형적입니다.
따라서 비선형 보정은 필요하지 않습니다.

Q8. ΔT 기반 이벤트 감지는 어느 시점에서 처리했나요?

A. PLC → HMI로 데이터를 받은 후
HMI에서 이전 값(prev)과 현재 값(curr)을 비교하여 ΔT를 계산합니다.
PLC에 과도한 연산 부담을 주지 않기 위한 구조입니다.

■ UI/UX·아키텍처 설계 관련 질문
Q9. 사용자 예약 기능에서 예외 처리 구조는 어떻게 구성했나요?

A.

프론트엔드 → UI Level Validation

Flask → Input Schema Validation

HMI → 장비 상태 기반 Validation

PLC → 모드/창고 상태 기반 Validation

총 4단계 계층 검증 구조입니다.

Q10. 관리자/사용자 UI를 분리한 구조적 이유는?

A.

접근 권한(Authorization)

정보 접근량(관리자 > 사용자)

조작 가능 기능(User: 예약, Admin: 승인/제어)

이 세 가지가 다르기 때문에
공통 UI를 사용하면 UX 혼란이 발생합니다.

■ 이벤트 시스템/장비 연계 관련 질문
Q11. 정전 이벤트 발생 시 PLC와 UI의 동기화는 어떻게 처리되나요?

A. PLC에서 PowerFlag = 0으로 변경 →
HMI ADS polling 감지 →
Flask 서버로 전달 →
웹 Dashboard에 실시간 반영됩니다.
이 상황에서 UI는 강제로 Operation Mode 제어를 비활성화합니다.

Q12. 지진 이벤트는 어떤 기준으로 판정하나요?

A.

Raw 진동값이 Threshold 이상 지속

0.5초 이상의 시간 안정성 확보

단발성 노이즈는 이벤트로 처리하지 않기 위해
0.5초 duration 조건을 추가했습니다.

■ 확장성·보안 관련 질문
Q13. 실시간성 요구가 높은데, 왜 WebSocket을 적용하지 않았나요?

A. PCST 장비 자체의 센서 갱신 주기가 느리기 때문에
WebSocket의 장점을 활용하기 어렵습니다.
Polling 방식이 오히려 구조적으로 단순하며
안정성·호환성이 높습니다.

Q14. 인증·보안 요소는 어떻게 구성했나요?

A.

세션 기반 간단 인증

Flask 데코레이터로 관리자/사용자 Role 분리

IP 기반 접근 제한

내부 네트워크 전용 모드로 구성

실제 산업용 수준으로 확장한다면
JWT + HTTPS + RBAC 권한 구조를 적용할 예정입니다.

★ 2. 팀원별 역할 기반 Q&A
■ (팀장) 이세종 — UI/시스템 구조 담당
Q. UI 구조를 설계할 때 가장 고려한 점은 무엇인가요?

A. 데이터가 3초 단위로 갱신되기 때문에
UI 레이아웃은 정보 밀도가 높고
상태 변화를 즉시 파악할 수 있어야 했습니다.
따라서 창고별 카드 UI + 알림 강조 표시 구조를 택했습니다.

Q. 관리자/사용자 화면을 분리한 이유는?

A. 권한·기능·데이터 접근량이 다르기 때문입니다.
특히 관리자는 이벤트 대응 기능이 있어
UI 복잡도가 훨씬 높습니다.

■ (이벤트 로직) 손석환 — PLC–서버 연결/이벤트 시스템 담당
Q. 지진·정전 이벤트는 어떤 흐름으로 처리되나요?

A. PLC 감지 → HMI 전달 → Flask 서버 전달 → 웹 팝업 표시
이 전체 흐름을 제가 구축했습니다.
이벤트 발생 시 UI에서 즉시 알람을 표시하도록 했습니다.

Q. 이벤트 처리의 어려웠던 점은?

A. 센서값 노이즈 때문에
이벤트가 잘못 발생하는 문제를 해결하는 것이 핵심이었습니다.
그래서 duration 기반 안정화 조건을 추가했습니다(예: 0.5초 유지).

■ (로그 시스템) 최현규 — 이벤트 로그/사용자 로그 담당
Q. 로그 시스템의 설계 목표는 무엇인가요?

A. 단순 기록이 아니라
후속 분석, 문제 재현이 가능하도록
모든 로그를 구조화된 JSON으로 저장하는 것이 목표였습니다.

Q. 왜 사용자 로그/관리자 로그/이벤트 로그를 분리했나요?

A. 각각의 목적이 다르기 때문입니다.
이벤트 로그는 장비 이상 분석,
사용자 로그는 행동 분석,
관리자 로그는 승인 이력 추적용입니다.

■ (사용자 기능) 김경은 — 입고 예약/날짜 설정/냉장·냉동 분류
Q. 보관 예약 기능 구현 시 가장 어려웠던 점은?

A. 날짜/창고 타입/물품 종류 검증을
클라이언트–서버–PLC에서 각각 따로 수행해야 한다는 점입니다.
입력 Validation을 3단계 구조로 설계하여 해결했습니다.

Q. 냉동·냉장·일반 창고 선택 기능은 어떤 구조인가요?

A. 사용자가 선택하면 서버에 요청이 전송되고
관리자는 승인 여부를 결정합니다.
승인되면 PLC 재고 상태가 자동 업데이트됩니다.

★ 3. 발표 중 받을 수 있는 ‘즉흥 질문’ 대비 답변 세트
Q. 이 프로젝트에서 가장 어려웠던 기술적 난점은 무엇인가요?

A. PLC–HMI–서버–웹이 모두 실시간으로 동작해야 해서
데이터 불일치(Inconsistency) 문제를 해결하는 것이 가장 어려웠습니다.
이 부분을 polling 지연 조절 + 이벤트 연동 구조 개선으로 해결했습니다.

Q. 데이터가 끊기면 어떻게 처리되나요?

A. HMI에서 통신 확인용 Heartbeat 변수를 추가해
서버로 전달되지 않으면
서버가 “Disconnected” 상태로 기록합니다.

Q. 팀원 간 역할 분배는 효율적이었나요?

A. 팀원별로 명확한 책임 범위를 설정했기 때문에
충돌 없이 병렬적으로 개발할 수 있었습니다.

Q. 시간을 더 준다면 어떤 개선을 가장 먼저 하겠습니까?

A. WebSocket 기반 실시간 알림과
MySQL 기반 로그 저장 시스템 도입입니다.

Q. 이 시스템은 실제 산업 적용이 가능한가요?

A. 구조적으로 가능합니다.
센서 수집–제어–웹 관제 구조가
실제 SCADA/HMI 시스템과 동일한 형태입니다.
